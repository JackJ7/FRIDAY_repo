## D2. Grounding contract ŌĆö the ground truth guaranteed every turn

The principle, stated once: **live authoritative sources outrank anything the model "remembers," and the code ŌĆö not the prompt ŌĆö decides what is authoritative.** Retrieved notes are *hints*; the clock, the calendar API, and the tracker files are *truth*. Today this holds for time and is partial for calendar. The contract makes it uniform.

**What is guaranteed in the system prompt every message (already true ŌĆö keep):**
- **Datetime.** `Current local time` + `The next 7 days` table (`engine.py:210`). Machine-local, DST-correct via the OS. *Add a one-line assertion that ISO dates come only from this table or a tool result ŌĆö never from memory.*
- **Invariants** from code (`invariants.py`, injected `engine.py:192`) ŌĆö cannot be dropped by any file edit.
- **Persona** (character + operating rules + voice head), read fresh each message so Obsidian edits take effect next turn (`engine.py:99,115`).
- **Accountability state** (open/overdue commitments, staleness) when wired (`engine.py:163`).

**What must become authoritative-by-rule (the gap ŌĆö Symptom 2):**
- **Calendar.** Introduce the same discipline the timelines already have. `operating_rules.md:141` says "When Jack asks about a timeline or its dates, call `read_timeline` FIRST, every time ŌĆö your conversational memory of dates goes stale." **Add the exact analogue for calendar:** *when Jack asks about an event, a meeting, or any date on the calendar, call `read_calendar` FIRST ŌĆö a note that mentions an event is not the calendar, and your memory of a date is not evidence.* This is a one-paragraph note edit, no code, and it directly kills the "scheduled for today / returns office hours" failure.
- **Task-DB / commitments.** Same rule already exists for timelines; extend the phrasing to commitments so "what am I supposed to do this week" hits `list_commitments`, not retrieval.

**How truth stays authoritative over memory (code-level, not vibes):**
1. **Retrieval floor (Symptom 6).** `KeywordRetriever.retrieve` must drop matches below a score threshold and, when the top hit is weak, return nothing rather than a low-relevance note. A note that survives arrives with its existing hedge *and* a provenance label (D7).
2. **Notes are never a source for dates.** The calendar-first rule plus a grounding self-check (D8): if a reply states a specific date/time for a calendar event and no `read_calendar`/`read_timeline` ran this turn, that is flagged in the log and (Phase 2) triggers a corrective pass ŌĆö the same pattern as the conjunct-completion floor (`engine.py:426`).
3. **One clock, one calendar-parse.** Both already enforced (`engine.py:210`, `calendar_sense._parse_start`). No second parse path may be added ŌĆö a second offset application is the original bug (`calendar_sense.py` docstring).

**Net:** the contract is ~90% built for time, needs a two-line rule for calendar authority, and needs the retrieval floor. Small, high-leverage, no architecture change.

## D3. Conversation-state design ŌĆö history, referents, working memory

**What exists (verified):** `self.history` holds this session's user/assistant/tool messages, trimmed to the last 40 (`engine.py:34-35, 464-466`); per-turn system guidance (referent block, voice) is deliberately **not** persisted so a stale copy can't fight the fresh one (`engine.py:463`). A **referent stack** records every tool-touched artifact/entity, most-recent-first, bounded to 12, with a content excerpt for files, and injects a resolution-rules block at the end of the system prompt (`engine.py:654-752`). Multi-part requests get a checklist injected up front and a completion floor after (`engine.py:282, 426`).

**The gap (Symptom 3):** the referent stack only tracks the six tool sources in `_REFERENT_SOURCES` (`engine.py:654`) ŌĆö files, notes, web pages, projects, playbooks. **A calendar event or a person named only in conversation is invisible to it.** When Jack said "the EPP MathWorks meeting" and then "an exact date," there was no entity on the stack to resolve against, so the 14B fell back to raw history and dodged. History *had* the turns; the model just didn't use them reliably, and nothing scaffolded the referent.

**The design:**
1. **Track conversational entities, not just artifacts.** When `read_calendar` returns events, push each event (title + local date) onto the referent stack as kind `event`; when `read_email`/`list_commitments` surface a named item, push it too. Then "that meeting", "the exact date", "when is it" resolve against a concrete stack entry with a real date ŌĆö exactly how "that document" already resolves. This is a small extension of `_track_referents` and `_REFERENT_SOURCES`, no new subsystem.
2. **Anti-dodge guard.** "Could you provide more context?" in reply to a *direct factual follow-up* is the named anti-pattern. Add it to the banned tells (D4) and, as the hard layer, a narrow code check mirroring the phantom-review barrier: if the user's message is a short follow-up referencing a stack entry (pronoun/deictic + no new subject) and the reply is a clarification request, regenerate once with "resolve the reference from the conversation; do not ask for context you already have."
3. **Leave history depth at 40 for now.** It is not the bottleneck ŌĆö the referent scaffold is. Revisit only if the D8 red-team dialogues show truncation dropping a still-relevant turn.

**Explicitly not doing:** summarizing/compressing live history mid-session. It adds a failure surface (lossy summary as false memory) for a problem the referent stack solves more cheaply. Cross-*session* continuity is a different mechanism ŌĆö that's D7.

## D6. Ask 2 ŌĆö GitHub write access (commit/push), confirmation-gated

**Reuse, don't reinvent.** The outbound pattern already exists and is the right home: `PermissionGate.approve_outbound` (`permissions.py:120`) confirms **every** outbound action with **no free tier and no allowlist**, and the confirm broker already blocks the engine worker on Jack's click (`service.py`, faceŌåöservice contract). Git write is a new *class* of outbound action that routes through this ŌĆö exactly as calendar-create does (`calendar_sense.create_event` ŌåÆ `approve_outbound`, `senses_tools.py:110` `kind="action_confirmed"`).

**What's genuinely new** is that git touches a **writable path outside brain/** ŌĆö and the current design *deliberately forbids that*: `repo_tools.py` is read-only by construction, the gate denies workspace writes, and `repo.allow_patches` is a `locked` config key documented as "the only future door" (`ARCHITECTURE.md:92-99`, CHANGELOG Task 5). So this ask **opens that door** and must do so carefully.

**The confirmation card shows (per the kickoff):** repo name + remote URL ┬Ę branch ┬Ę files changed (added/modified/deleted) ┬Ę a diff **stat + summary** (not the full diff ŌĆö preview like `permissions._describe`, `permissions.py:199`) ┬Ę the exact commit message ┬Ę and a bold flag if the push targets a **shared/remote** branch vs. a local one.

**Auto-blocked regardless of confirmation** ŌĆö a code-level deny layer *before* the card is ever shown, mirroring the gate's existing "deny zone" (`permissions._zone` returning `None`):
- **Force-push** (`--force`/`+refs`), **history rewrite** (rebase/amend of pushed commits, `reset --hard` on a pushed ref).
- **Protected branches** ŌĆö `main`/`master`/`release/*` (configurable, `locked` tier so she can't self-widen it).
- **Secrets in the diff** ŌĆö a scanner (high-entropy strings, key/token patterns, `.env`) blocks the commit and reports what it found. Reuses the same "code does the pattern part" instinct as `senses/importance.py`.
- **Push to a repo/branch not on Jack's allowlist** ŌĆö a `locked` config list of writable repos; anything else denies.

**Reuse vs. extend, precisely:** *reuses* `approve_outbound` + the confirm broker + the `locked`-tier config pattern (`config_governance.py`) for the protected-branch/allowlist knobs. *Extends* with (1) a new `git_commit_push` tool (`kind="action_confirmed"`), (2) a pre-confirmation hard-block layer (the auto-blocks above), (3) a new writable zone for the git worktree governed entirely by that layer ŌĆö **not** a general loosening of `permissions._zone`. Taint interaction is free: the tool is `external_read`-adjacent (repo content is outside the trust boundary), so a diff assembled from tainted content still forces the confirm. Net: a contained, gate-consistent capability that cannot become a silent write path.

## D7. Ask 4 ŌĆö claude-mem-equivalent memory (the backbone of Ask 3)

**What FRIDAY has (verified).** A post-exchange **memory pass** that reviews each turn and commits durable facts to the authoritative note, told the ground-truth write ledger so it can't be fooled by a reply that only *claims* to have saved (`engine.py:795-951`). Write guards that preserve hand-written content (read-before-overwrite, `brain.py`). A **session greeting** built from the five most-recently-edited notes (`engine.py:955`). Interaction JSONL logs of every turn. A clean `Retriever` abstraction (`retriever.py`) with a keyword implementation behind it.

**What's missing vs. claude-mem ŌĆö the real gap (Symptom 8):** there are **no typed observations** (categorized decision/bugfix/discovery records with IDs, timestamps, titles), **no provenance IDs** on recalled facts, **no 3-layer progressive retrieval** (index ŌåÆ IDs ŌåÆ fetch), and **no cross-session working continuity** beyond "durable facts became notes" + "greeting reads recent notes." `self.history` is in-memory and dies at session end (`engine.py:34`). **The decision ŌĆö adopt vs. wrap vs. reimplement:**
